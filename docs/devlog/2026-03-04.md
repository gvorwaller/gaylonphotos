## 14:30 EST - Fix Map Marker Click Regression (Google Maps API Change)

### Summary
Google Maps API silently deprecated `addListener('click')` on AdvancedMarkerElement overnight, breaking all marker InfoWindow popups. Diagnosed with claude-in-chrome browser automation, fixed with dual event listener strategy + lazy InfoWindow creation.

### What Was Done
- User reported ancestry diamond markers no longer showing InfoWindow popups on click
- Used claude-in-chrome to reproduce: physical clicks on markers produced no popup
- Console revealed: `<gmp-advanced-marker>: Please use addEventListener('gmp-click', ...) instead of addEventListener('click', ...)`
- Discovered `addEventListener('gmp-click')` fires programmatically but NOT from physical clicks on custom content elements
- Added `contentEl.addEventListener('click', handleClick)` directly on the marker's DOM element — this reliably catches physical clicks
- Also discovered InfoWindow was null after Vite HMR: `$state(map)` survives HMR but plain `let infoWindowInstance` resets, and `if (map) return;` guard skips re-creation
- Fixed with lazy InfoWindow creation in the click handler
- Build passed, committed, pushed, deployed to prod — gaylon.photos returning 200

### Files Modified
- `src/lib/components/common/Map.svelte` — event binding, lazy InfoWindow, cleanup restructured

### Key Technical Insight
- Google Maps AdvancedMarkerElement with custom `content` elements: `gmp-click` does NOT fire from physical clicks on the content div. Must add a standard DOM `click` listener on the content element itself
- Vite HMR preserves Svelte 5 `$state` variables but resets plain `let` declarations. If init logic is guarded by `if (stateVar) return;`, non-reactive variables set inside that guard will be permanently lost after HMR. Fix: lazy-create or use `$state` for critical instances
- Previous devlog note "addListener('click') works; addEventListener('gmp-click') does not fire" was correct at the time — Google Maps API changed behavior between sessions
