# 2026-03-02 Devlog

## 01:30 EST - Ancestry Feature: GEDCOM Parser Fix + Browser Testing

### Summary
Fixed a GEDCOM parser bug discovered during browser-based import testing, where family tree traversal stopped at the root person because FAMC/FAMS links weren't being inferred from FAM records.

### What Was Done
- Generated test GEDCOM data (10 people, 4 generations, Oslo/Drammen/Gothenburg locations)
- Uploaded via browser-authenticated fetch to POST /api/ancestry
- Discovered only 1 person imported (root) — BFS couldn't traverse family links
- Root cause: GEDCOM parser relied on FAMC tags on INDI records, but many GEDCOM files only define relationships in FAM records (HUSB/WIFE/CHIL)
- Fix: Added post-parse linking step in `parseGedcom()` that infers FAMC/FAMS from FAM records
- Re-tested: 9 of 10 persons imported correctly (10th excluded as not in direct lineage — correct behavior)
- Geocoding: 5 ok, 2 approximate, 1 failed (Tanum without umlaut)
- Verified all public UI: collapsible panel, all 3 tabs (By Place, By Family Line, By Generation), map markers (purple diamonds), stats footer, FamilySearch links
- Added ancestry.json and test-data/ to .gitignore

### Files Modified
- `src/lib/server/ancestry.js` — post-parse FAMC/FAMS inference from FAM records
- `.gitignore` — exclude ancestry.json data files and test-data/

### Key Technical Insight
GEDCOM files from FamilySearch include FAMC/FAMS on INDI records, but many other exporters only define relationships in FAM records. The post-parse linking step makes the parser robust to both styles. This was caught immediately by real import testing.
