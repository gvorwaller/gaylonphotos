## 20:00 EST - Swift CLI Photo Uploader: PHImageManager Deadlock (Unresolved)

### Summary
Built a native Swift CLI (`mac/photo-uploader-cli`) to export photos directly from Photos.framework and upload to the backend. Preflight, auth, and selection reading all work. Export hangs due to PHImageManager threading issue.

### What Was Done
- **Built complete Swift Package** with ArgumentParser: preflight checks, API auth, export+upload pipeline
- **Preflight works**: macOS version, Photos running, library access, selection count, asset classification, backend reachability — all pass
- **Auth works**: successfully authenticates to gaylon.photos with admin credentials
- **Selection reading works**: AppleScript gets media item IDs, PHAsset.fetchAssets resolves them
- **Export hangs**: `PHImageManager.default().requestImageDataAndOrientation()` never returns/completes in the CLI context

### Approaches Tried (All Failed)
1. **Async callback + semaphore on main thread** — deadlock, main thread blocked
2. **`isSynchronous = true` on main thread** — deadlock, Photos.framework needs main thread internally
3. **`isSynchronous = true` on `DispatchQueue.global` + semaphore** — still hangs
4. **Pipeline on background thread + main RunLoop pumping** — still hangs
5. **Standalone `swift -` script with same pattern** — this WORKS (returns 523KB JPEG), proving the API itself is functional

### Key Finding
The exact same PHImageManager code works in a standalone Swift script but hangs in a compiled Swift Package CLI. The difference is likely in how the process's RunLoop/threading infrastructure is set up. This is not a simple threading fix — it requires understanding Photos.framework's internal dispatch requirements in a CLI context vs. an interpreted script context.

## 23:30 EST - Continued Debugging: NSApplication + PHAssetResourceManager (Still Unresolved)

### Summary
Deep research into Photos.framework internals led to two architectural changes, but the real blocker turned out to be shared album (`typeCloudShared`) assets — not threading. Handing off to Codex.

### Research Findings
- **PhotosExporter** (the only working open-source Swift photo export tool) uses `@NSApplicationMain` — it's an NSApplication-based app, not a plain CLI
- **Apple Developer Forums + objc.io** confirm: `RunLoop.main.run()` is NOT sufficient for AppKit-dependent frameworks; `NSApplication.shared.run()` is required
- Standalone compiled test (`swiftc` binary, not SPM) with `NSApplication.shared.run()` + `PHAssetResourceManager.writeData` exports a **local** photo in 0.0 seconds — architecture proven correct

### What Was Done
1. **Converted CLI to NSApplication-based tool** — added `main.swift` with `NSApplication.shared`, `setActivationPolicy(.accessory)`, `AppDelegate` pattern. Removed `@main` from `ParsableCommand`. All interactive prompting happens before `app.run()`.
2. **Switched from PHImageManager to PHAssetResourceManager** — `PHImageManager.requestImageDataAndOrientation` deadlocks internally (its own semaphore in `PHImageRequest.startRequest` blocks, confirmed via `sample` stack traces). `PHAssetResourceManager.writeData` avoids this entirely.
3. **Process sampling** revealed the exact deadlock: with `isSynchronous=true`, Photos.framework's internal `PHImageRequest.startRequest` uses a dispatch semaphore that waits for a callback that never arrives. With `isSynchronous=false`, Photos.framework returns immediately but never delivers the callback.

### The Real Blocker: Shared Album Photos
All 5 selected test photos are `sourceType: .typeCloudShared`. Stack trace analysis and testing proved:
- **Local library photos**: `PHAssetResourceManager.writeData` works instantly (0.0s, 2.9MB)
- **Shared album photos**: `PHAssetResourceManager.writeData` callback never fires. `PHImageManager` also hangs. AppleScript `export` also hangs (iCloud download stalls).
- The user can view and export these same photos from Photos.app UI in <10 seconds — Photos.app uses private APIs that external processes cannot access for shared album data.
- This is a **known PhotoKit limitation**: `.typeCloudShared` image assets have resources listed but data is not accessible through public PhotoKit APIs.

### Current State of Code
- `mac/photo-uploader-cli/Sources/main.swift` — NSApplication entry point with AppDelegate
- `mac/photo-uploader-cli/Sources/PhotoUploader.swift` — ParsableCommand struct (no @main), execute() method
- `mac/photo-uploader-cli/Sources/ExportUploadPipeline.swift` — dual-path export: PHAssetResourceManager for local, AppleScript for shared
- `mac/photo-uploader-cli/Sources/APISession.swift` — auth + upload (working)
- `mac/photo-uploader-cli/Sources/Preflight.swift` — preflight checks (working, but iCloud detection is wrong — reports shared as "local")
- Architecture is sound; local photos upload end-to-end. Shared album export is the unsolved problem.
