### Summary
Built the complete AI-powered photo geolocation feature using Gemini 2.5 Flash, ran a full 130-photo live test on `scandinavia-2023`, discovered and fixed two critical bugs in the save flow (triple-fire and state reset), and created a follow-up task for itinerary-based location fallback.

### What Was Built

**Server-side (Gemini integration):**
- `src/lib/geovision-prompt.js` ŌĆö shared prompt and response parser. Gemini is instructed to return JSON with `{lat, lng, placeName, reasoning, confidence}`. Confidence tiers: high/medium/low/none. Parser handles markdown fences and bare JSON.
- `src/lib/server/geovision.js` ŌĆö fetches image from CDN URL, converts to base64 inline data, sends to Gemini 2.5 Flash (`@google/generative-ai`). Returns parsed location or `{ _rateLimited: true }` on 429s.
- `src/routes/api/geovision/+server.js` ŌĆö POST endpoint with two actions:
  - **identify** (default): preview-only, sends each photo to Gemini, returns results without saving. Max 50 photos/request.
  - **apply**: saves approved locations with `gpsSource: 'ai'`, `locationAI: { model, detectedAt }`.

**Client-side (review panel):**
- "Auto-locate N photos" button on admin collection page for photos without GPS
- Confirmation modal before starting (shows photo count)
- Sequential processing with live progress counter (~5s per photo)
- Results grouped into review panel:
  - **Located** (green/yellow/red by confidence): shows thumbnail, filename, place name, reasoning, map preview
  - **Failed/No-location**: correctly identified as unlocatable (wildlife close-ups, etc.)
  - **Skipped**: already has GPS
- Approve/reject toggles: HIGH auto-approved, MEDIUM/LOW default unchecked
- "Save N Approved Locations" button ŌåÆ batch save via apply action

### Bugs Found and Fixed

**Bug 1: Triple-fire on save button (lost 55 locations)**
- Console showed `saveApprovedLocations` called 3├Ś at identical timestamp
- Root cause: `autoLocateSaving` is `$state` ŌĆö Svelte 5 batches reactive updates, so multiple synchronous reads all see `false` before any write takes effect within the same tick
- Fix: plain `let _saveLock = false` (non-reactive) checked at function entry ŌĆö synchronous guard prevents re-entry

**Bug 2: State reset after save (count showed 130 instead of 75)**
- After saving, the page still showed "Auto-locate 130 photos" ŌĆö the saves worked server-side but the client didn't reflect them
- Root cause: `$effect` at line 12 re-syncs `photos` from `data.photos` on any load data change, but `data.photos` was never invalidated after the save
- Fix: `await invalidateAll()` after successful save forces SvelteKit to re-run the load function, fetching fresh photo data from disk
- User feedback that led to the fix: "WHY is that 'fine'. That is not 'fine'. It's broken." ŌĆö correctly identified the state reset as a real bug, not acceptable HMR behavior

### Files Modified

### Key Technical Insight
Svelte 5 `$state` variables cannot be used as synchronous mutex guards. Because reactive updates are batched, reading a `$state` variable immediately after setting it in another call returns the old value within the same tick. For function-level re-entry guards, use a plain `let` variable ŌĆö it updates synchronously and is visible to all callers immediately.

Separately, SvelteKit's `data` object from load functions is a snapshot ŌĆö mutations to server state via API calls don't automatically propagate back. `invalidateAll()` is the correct way to force a re-fetch after server-side writes that change the data the load function returns.

## 20:00 EST - GEDCOM Re-Import Merge: Initial Implementation (CC3)

### Summary
Implemented the full GEDCOM re-import merge feature (td-8dfca1) from the design doc (`docs/2026-03-21_gedcom-reimport-merge-design.md`). All four phases ŌĆö override tracking, diff engine, preview UI, and apply merge ŌĆö were built in a single session. CODEX (12:40) and CC4 (16:00) then reviewed and hardened the code across 5 adversarial rounds + cross-review.

### What Was Built (Pre-Review)

**Phase A ŌĆö Override Tracking:**
- `updatePersonField()` in `ancestry.js` ŌĆö creates `appOverrides` entries on person records, syncs year fields, rebuilds places
- Extended `PATCH /api/ancestry` to detect `personId + field` body shape for person edits (vs existing `placeId` for coordinates)
- Person edit UI in `AncestryEditor.svelte` ŌĆö pencil icon per person row, inline edit panel (6 fields), optional reason, override badges, revert-to-original

**Phase B ŌĆö Diff Engine:**
- `diffAncestry()` in `ancestry.js` ŌĆö matches persons by fsId (primary) ŌåÆ xref ID (fallback) ŌåÆ name+birthYear (fuzzy, flagged)
- Field-by-field comparison for name, gender, birthDate, birthPlace, deathDate, deathPlace
- Fact comparison by composite key (type|date|place)
- Auto-rejects GEDCOM changes on fields with existing `appOverrides`
- Returns structured diff: summary, changes[], added[], removed[]

**Phase C ŌĆö Preview UI + API:**
- `POST /api/ancestry/reimport` ŌĆö preview endpoint (multipart upload, returns diff)
- `PUT /api/ancestry/reimport` ŌĆö apply endpoint (JSON with decisions)
- New route: `/admin/[collection]/ancestry/reimport` with page server loader
- `AncestryReimport.svelte` ŌĆö 3-step flow:
  1. Upload with person picker + max generations slider
  2. Diff review with expandable accordions, per-field accept/reject toggles, bulk actions, match-type badges (fsId/xref/fuzzy), override protection indicators (lock icon)
  3. Apply with confirmation modal showing summary

**Phase D ŌĆö Apply Merge:**
- `applyReimport()` in `ancestry.js` ŌĆö re-parses GEDCOM, re-runs diff, applies accepted field changes, adds new persons (with geocoding), removes confirmed deletions
- Preserves GPS/geocode data and app overrides on rejected fields
- Recomputes generation/lineage from new tree structure
- Rebuilds places array with preserved geocoding
- Records `mergeHistory` entry with type "reimport"

### Architecture Decisions
- **Match strategy**: fsId is the primary key (stable across GEDCOM exports from FamilyTree 11). Xref and name+year are fallbacks, with fuzzy matches flagged in the UI.
- **Override protection**: Fields with `appOverrides` default to "reject" in the diff, showing the GEDCOM value but not applying it unless the user explicitly accepts.
- **GPS preservation**: Place coordinates, geocodeStatus, and nearStop are always app-owned ŌĆö never overwritten by GEDCOM re-import.
- **Server-side GEDCOM cache**: Stores parsed GEDCOM text server-side with a `previewId` token, avoiding 10MB roundtrip through the client between preview and apply steps. 30-min TTL, max 5 entries.
- **Atomic apply**: `applyReimport` uses `updateJson` with a sync callback ŌĆö re-diffs against the locked file to handle concurrent edits between preview and apply.
