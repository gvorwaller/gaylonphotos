### Key Technical Insight
- **Backward compatibility via poster JPEG**: The `url` field on video records points to a poster JPEG (not the video), so every existing `<img src={photo.url}>` works unchanged. The new `videoUrl` field is opt-in. This made the rollout safe across all components.
- **HEIC vs MP4 share `ftyp` magic bytes**: Both use ISO BMFF container format. Disambiguation requires checking the brand at bytes 8-12 (`isom`/`mp41`/`qt  ` = video, `heic`/`mif1` = image).
- **iPhone HEVC Live Photo clips**: 0.57-second HEVC clips with bt709 color space fail ffmpeg's MJPEG encoder. Fixed with `-pix_fmt yuvj420p` flag and a PNG fallback path for edge cases.
- **Apple Photos exports duplicates**: Same video appears as both .MOV (original) and .MP4 (compatible). Must deduplicate before ingest.

## 19:10 EST - Time Machine Failure Analysis and Backup Strategy Notes

### Summary
Investigated why the Mac's Synology Time Machine backup was failing with "not enough room" even though DSM trash deletion was enabled. Confirmed the issue is not DSM trash behavior: `backupd` is pruning old backups, but the `1 TB` Synology Time Machine allocation is already full enough that thinning only reaches about `99.0-99.5 GB` free while macOS currently wants `100 GB` minimum free space to begin the next backup.

### What Was Found
- Active Time Machine destination is `smb://M2user@192.168.22.74/TimeMachine`
- Local `backupd` logs on `2026-04-02` show:
  - `Insufficient free space after thinning all available backups - free space: 99.5 GB, minimum free space: 100 GB`
  - `BACKUP_FAILED_TARGETVOL_DISK_FULL (56)`
- Time Machine preferences show:
  - `BytesUsed = 1028481863680`
  - `BytesAvailable = 99036860416`
  - `InheritanceDecision = 3`
  - multiple `HostUUIDs`
- Mounted backup volume name indicates the destination is likely carrying inherited history from an older machine:
  - `Backups of Gaylon's MacBook Air M2 2022`
- Current live user data on the M4 looks far smaller than the used backup set, which reinforces that retained history/inheritance is the space consumer rather than current disk usage alone

### Interpretation
- Time Machine pruning is working
- DSM recycle bin settings are not the root cause
- The likely problem is an inherited backup chain from the older M2 continuing inside the same `1 TB` Time Machine allocation
- The failed vacation window likely mattered only because the Mac returned with a larger delta to back up; pruning still could not recover enough headroom to satisfy backupd's current minimum free-space requirement

### Recommended Immediate Fix
- Remove or disable the existing Time Machine destination on the Mac
- Verify no backup is running
- On Synology, delete only the contents of the dedicated `TimeMachine` shared folder
- Empty DSM recycle bin for that share so the space is actually reclaimed
- Re-add the Time Machine destination and let it seed a fresh full backup

### Longer-Term Recommendation
- If continuing with Time Machine on a fixed `1 TB` quota, expect this to recur eventually; Time Machine is a history system, not a simple mirror
- Better fit for this environment:
  - `Arq` backing up the Mac to Synology
  - optional Synology `Hyper Backup` to DigitalOcean Spaces for offsite copy of NAS data
- Arq pricing checked during this investigation:
  - `Arq 7` is currently `$49.99` one-time per computer, with `12 months` of updates included

### Key Technical Insight
The misleading part of this failure is that the Mac's live disk usage can remain comfortably below the Synology allocation while Time Machine still fills the destination. Once inherited history from an older machine is mixed into the same target, the quota is consumed by long-lived retention rather than by current working data.
